# Tool-Friendly Compiler Design for Perseus

## Introduction

This document captures compiler-engineering practices that make Perseus easier to maintain, inspect, debug, and integrate with developer tools.

The emphasis here is not on making Perseus "AI-first" or claiming that it is uniquely suited to automated reasoning. Instead, the focus is on broadly useful qualities:

- structured diagnostics with stable identifiers
- deterministic and reproducible output
- clear internal stage boundaries
- useful debug metadata and source mapping
- fast feedback loops for editing and testing

These qualities benefit human developers first, while also making future integrations with IDEs, static analyzers, language servers, and automated tooling more practical.

## Compiler Architecture Overview

A tool-friendly compiler pipeline should be organized into clear, inspectable stages. In Perseus today, some of these stages already exist explicitly, while others remain possible future refactor directions.

Current and proposed stages:

1. **Lexing**: Tokenizes Perseus source code.
2. **Parsing**: Produces a parse tree.
3. **Name Resolution / Symbol Collection**: Resolves identifiers, scopes, declarations, and procedure metadata.
4. **Type Inference / Type Checking**: Annotates expressions with resolved types.
5. **Code Generation**: Emits Jasmin assembly.
6. **Class File Production**: Assembles Jasmin into JVM `.class` files.
7. **Post-Processing / Verification**: Recomputes frames or verifies generated bytecode where needed.

Possible future stages:

8. **AST or IR Generation**: Introduce an explicit intermediate representation for analysis or tooling.
9. **IR-Level Optimization**: Perform more structured lowering or optimization before Jasmin emission.

## Why Stage Separation Matters

Clear stage boundaries improve the compiler even without any advanced tooling:

- Easier debugging and more focused tests
- Better separation of concerns between parsing, typing, and code generation
- More stable behavior as features are added
- Cleaner opportunities for future IDE, diagnostics, or language-server support

Perseus already follows this principle in part through its symbol-table pass, type-inference pass, code-generation phase, and post-generation verification helpers.

## Diagnostic Design

Diagnostics are one of the most important interfaces between the compiler and its users. Good diagnostic design should aim for:

The current implementation status and active error-code set are documented in [Compiler Diagnostics.md](Compiler%20Diagnostics.md).

- **Stable Error Codes:** Assign unique, stable identifiers to diagnostics so they can be referenced in docs, tests, and tooling.
- **Deterministic Ordering:** Emit diagnostics in a predictable order, such as by file, line, and column.
- **Multiple Errors per Run:** Report as many independent issues as possible instead of aborting after the first failure.
- **Useful Source Span Information:** Include file name, line, and column for the relevant source span.
- **Actionable Suggestions:** Where practical, provide concrete hints or suggested fixes.

Possible future enhancement:

- **Structured Diagnostic Output:** Add JSON or another machine-readable format for diagnostics so editors and tools can consume compiler results more directly.

## Deterministic Code Generation

Deterministic output is valuable for reproducible builds, stable tests, and easier debugging. Key practices include:

- **Deterministic Label Naming:** Use predictable label schemes based on source order or structured counters.
- **Stable Ordering:** Emit methods, fields, and synthetic classes in a canonical order.
- **Canonical Formatting:** Format Jasmin output consistently.
- **Reproducible Output:** Ensure identical inputs produce identical emitted Jasmin and, where practical, identical class files.

These are useful engineering goals regardless of whether the consumer is a human reading generated Jasmin, a test suite comparing outputs, or an external tool.

## Stack and JVM Verification Safety

JVM bytecode must satisfy strict verification rules. Compiler design should aim to make verifier failures rare, diagnosable, and testable.

Important practices:

- Track stack behavior carefully during code generation.
- Compute or conservatively approximate `.limit stack` and `.limit locals`.
- Verify generated bytecode as part of tests.
- Map verifier failures back to the generated source or original source program as clearly as possible.

Perseus currently uses a combination of conservative limits and ASM-based post-processing/verification. That is a practical strategy, but it still leaves room for future improvement in precise limit calculation and better error reporting.

## Debug Metadata and Source Mapping

To support debugging and runtime analysis:

- Emit line information where practical so generated bytecode can be related back to source.
- Preserve meaningful variable names and scopes when possible.
- Maintain a path from generated artifacts back to source-level constructs.

Possible future directions:

- richer `.line` generation
- more systematic source-span tracking through code generation
- clearer mapping of JVM verification/runtime failures back to Perseus source locations

## Intermediate Representations

Perseus does not currently require a formal AST or JVM IR in order to function, but an explicit intermediate representation may become valuable as the language grows.

Potential benefits of a future IR include:

- simpler lowering from high-level language constructs
- easier analysis and transformation
- cleaner separation between front-end semantics and JVM-specific emission
- better support for future tooling, refactoring, or optimization work

This should be treated as a design option, not a claim about the current implementation.

## CLI and Tooling Design

A tool-friendly compiler should offer commands and outputs that support both everyday use and inspection.

Useful long-term CLI capabilities include:

- **check**: Parse and validate source files without producing runnable output
- **build**: Run the full compilation pipeline
- **emit-jasmin**: Output generated Jasmin for inspection
- **assemble**: Assemble existing Jasmin into `.class` files
- **run**: Compile and execute a program in one step
- **emit-ast / emit-ir**: Output structured intermediate representations if those stages are introduced later

Fast validation workflows are especially important. Even if Perseus never grows a large tooling ecosystem, a quick "check" mode would still improve day-to-day compiler development.

## Testing Strategy

Testing should reinforce correctness, stability, and inspectability:

- **End-to-End Runtime Tests:** Compile and run sample programs, then assert on outputs or behavior.
- **Golden Tests for Diagnostics:** Compare diagnostics to expected results once diagnostics become more structured.
- **Snapshot Tests for Generated Jasmin:** Compare emitted assembly where stable output matters.
- **Verification Tests:** Confirm generated bytecode passes JVM/ASM verification.
- **Deterministic Output Checks:** Ensure stable generation when reproducibility matters.

Perseus already relies heavily on end-to-end runtime tests. The other categories can be added selectively as the compiler matures.

## Long-Term Considerations

Useful future directions include:

- versioned structured diagnostic schemas
- explicit IR formats if new intermediate stages are added
- language-server-friendly compiler output
- a small, predictable runtime surface for generated programs

None of these are prerequisites for Perseus today, but they are good design pressures to keep in mind as the compiler evolves.

---

The main idea is simple: a compiler that is deterministic, debuggable, and explicit is easier to extend and easier to trust. Those qualities make Perseus better for both human developers and future tooling, without requiring the project to define itself around AI claims.
